# 阶段一：指挥中枢实施记录

## 1. 阶段目标

第一阶段的目标不是更换解析引擎，而是先把知识库文档入库链路从“单一路径硬编码”整理成“可路由、可标记、可扩展”的结构，为第二阶段 QMD、第三阶段 MinerU / Docling 留出稳定接入口。

本阶段完成的核心目标：

1. 打通 `docType`，让前端、接口、数据库、日志、向量 metadata 使用同一套文档类型语义。
2. 新增轻量编排中枢，把 `KnowledgeDocumentServiceImpl` 中的解析/切块/向量化主链路抽离出去。
3. 把 metadata 写入和 metadata 过滤闭环做通，让后续命名空间检索可以直接复用。
4. 保持第一阶段保守策略不变：`academic_paper / project_report` 仍先走 Tika，`general` 改为 MIME 路由。
5. 补齐可观测字段和入口过滤能力：文档、分块日志、向量 metadata、聊天入口都能看到或使用 `docType / parseEngine / chunkEngine`。

---

## 2. 本阶段实际改造内容

## 2.1 `docType` 贯穿前后端与数据库

新增了文档业务类型枚举：

- `general`
- `project_report`
- `academic_paper`

改造点如下：

- 后端请求对象新增 `docType`
- `KnowledgeDocumentDO / VO` 新增 `docType`
- `KnowledgeDocumentChunkLogDO / VO` 新增 `docType`
- 上传与编辑页面新增“文档类型”选择
- PostgreSQL schema 与升级脚本新增 `doc_type`

默认值统一为 `general`，这样老数据和未指定的文档都能平滑兼容。

## 2.2 新增摄取编排器，主链路不再直接写死解析过程

新增文件：

- `bootstrap/src/main/java/com/nageoffer/ai/ragent/knowledge/enums/DocumentType.java`
- `bootstrap/src/main/java/com/nageoffer/ai/ragent/knowledge/service/ingest/DocumentIngestionRoute.java`
- `bootstrap/src/main/java/com/nageoffer/ai/ragent/knowledge/service/ingest/KnowledgeDocumentIngestionOrchestrator.java`

现在的职责划分如下：

- `KnowledgeDocumentServiceImpl`
  - 负责上传、任务状态、日志、事务收口
- `KnowledgeDocumentIngestionOrchestrator`
  - 负责路由决策
  - 负责解析
  - 负责切块
  - 负责向量 metadata 注入
  - 负责 embedding 前的统一处理

这样做的直接收益是：

1. 第一阶段先把骨架搭起来，不改变整体业务流。
2. 第二阶段接 QMD 时，主改动集中在 orchestrator 和 chunk engine，不需要继续膨胀 `KnowledgeDocumentServiceImpl`。
3. 第三阶段接 MinerU / Docling 时，也只需要替换路由和解析实现。

## 2.3 第一阶段路由策略已落地

当前实际路由规则：

| `docType` | 解析器 | 说明 |
| --- | --- | --- |
| `general` | `selectByMimeType(fileType)` | 让 Markdown / text/plain 真正走 `MarkdownDocumentParser` |
| `project_report` | `Tika` | 第一阶段先保守兼容 |
| `academic_paper` | `Tika` | 第一阶段先保守兼容 |

这一步解决了一个关键历史问题：主链路此前始终写死 `Tika`，导致 `DocumentParserSelector.selectByMimeType()` 明明存在，却没有真正进入生产链路。

## 2.4 统一写入文档 metadata

在 `KnowledgeDocumentIngestionOrchestrator` 中，所有 `VectorChunk` 在向量化前都会统一注入 metadata：

- `kb_id`
- `doc_id`
- `doc_name`
- `doc_type`
- `source_type`
- `ingest_source_type`
- `file_type`
- `chunk_strategy`
- `chunk_engine`
- `chunk_fallback`
- `parse_engine`
- `parser_type`

其中：

- `source_type` 本阶段按重构建议，与 `doc_type` 保持同值，用于后续命名空间检索
- `ingest_source_type` 保留原有 `file / url` 语义，避免和业务类型混淆

此外，文档重新启用时的向量重建链路也补上了同样的 metadata 注入，避免出现“首轮入库有标签、重新启用后标签丢失”的问题。

## 2.5 检索层 metadata 过滤闭环完成

本阶段补齐了 `RetrieveRequest.metadataFilters` 的真实执行逻辑，并把过滤条件从聊天入口一路传到具体向量检索器：

- `PgRetrieverService`
  - 动态拼接 `metadata->>? = ?`
- `MilvusRetrieverService`
  - 动态生成 `metadata["key"] == value` 过滤表达式
- `RetrievalEngine / MultiChannelRetrievalEngine`
  - 透传 metadataFilters 到多通道检索上下文
- `VectorGlobalSearchChannel / IntentDirectedSearchChannel`
  - 透传 metadataFilters 到全局检索和意图定向检索
- `CollectionParallelRetriever / IntentParallelRetriever`
  - 构造 `RetrieveRequest` 时带上 metadataFilters

这意味着从本阶段开始，后续只要上层调用传入：

```json
{
  "doc_type": "academic_paper"
}
```

就已经具备按文档类型隔离检索的能力，不需要再返工底层向量检索实现。

## 2.6 parse/chunk 引擎状态持久化

为方便人工验收、灰度排障和后续阶段对比，本阶段补齐了最终使用引擎的持久化字段：

- `t_knowledge_document.parse_engine`
- `t_knowledge_document.chunk_engine`
- `t_knowledge_document_chunk_log.parse_engine`
- `t_knowledge_document_chunk_log.chunk_engine`

每次分块任务完成后，文档表和分块日志都会记录本次实际使用的解析器和切块器。QMD 回退等场景也可以通过 `chunk_engine` 和向量 metadata 中的 `chunk_fallback` 追踪。

## 2.7 聊天入口支持按文档类型检索

阶段一不只停留在“底层能过滤”，还补齐了用户侧入口：

- `GET /rag/v3/chat` 新增可选参数 `docType`
- 前端聊天输入区新增文档类型筛选
- `docType=general / project_report / academic_paper` 会被转换为 `metadataFilters.doc_type`
- 不传或选择“全部文档”时保持原有混合检索行为

## 2.8 前端文档管理页已同步

改造范围：

- 上传弹窗新增“文档类型”选择
- 编辑弹窗新增“文档类型”编辑
- 文档列表新增“文档类型”列
- 分块日志视图支持展示 `docType`
- `knowledgeService.ts` 补齐 `docType` 字段定义和接口入参

这样第一阶段已经形成完整链路：

`前端选择 docType -> 后端持久化 -> 编排器路由 -> metadata 落库 -> 聊天入口过滤 -> 检索过滤`

## 4.2 后端单元测试

### 测试前说明

仓库当前 `pom.xml` 中的 surefire 配置存在一个既有问题：

```xml
<argLine>@{argLine} -javaagent:...</argLine>
```

在默认 fork 模式下该占位符没有被展开，fork JVM 会直接失败。  
本次测试使用 `-DforkCount=0` 规避这一仓库既有问题，不属于第一阶段改造引入的问题。

### 执行命令

```powershell
.\mvnw.cmd -pl bootstrap -am "-Dtest=KnowledgeDeletionCleanupTest,KnowledgeDocumentIngestionOrchestratorTest,QmdProcessClientTest,PgRetrieverServiceTest,MilvusRetrieverServiceTest,ParallelRetrieverMetadataFilterTest" "-Dsurefire.failIfNoSpecifiedTests=false" "-Dspotless.apply.skip=true" "-Dspotless.check.skip=true" "-DforkCount=0" test
```

### 结果

- `BUILD SUCCESS`
- 测试总数：`15`
- 失败：`0`
- 错误：`0`
- 跳过：`0`

覆盖的测试点：

   - 验证文档删除与资源清理逻辑在新增 orchestrator 依赖后仍然正常
2. `KnowledgeDocumentIngestionOrchestratorTest`
   - 验证 `general` 文档走 MIME 路由
   - 验证 `academic_paper` 第一阶段回落 Tika
   - 验证 chunk metadata 注入
3. `PgRetrieverServiceTest`
   - 验证 PostgreSQL metadata 过滤 SQL 生成正确
4. `MilvusRetrieverServiceTest`
   - 验证 Milvus metadata 过滤表达式生成正确

   - 验证全局检索并行器会把 `doc_type` metadataFilters 写入 `RetrieveRequest`

第一阶段已经达到 100% 可验收状态，本轮在这里暂停，等待人工确认后再继续第二阶段：

4. `parseEngine / chunkEngine` 已进入文档表、分块日志和前端展示，便于人工验收与排障。
5. 前端、后端、数据库、测试与实施文档已经同步。
